## 0:00–1:00 · 开场(A)

[画面:README 顶部 / 或一张标题页]

大家好,我们这个项目叫 **Alpaca 系统化交易系统**,是 FINM 25000 的小组作业。简单说,我们从零搭了一套**端到端、可以真的跑起来**的量化交易系统:它会自动从 Alpaca 拉行情、按规则生成交易信号、经过风控之后自动下单,还有一个网页面板可以实时监控和控制。

先说一个最重要的前提:**我们全程只用 Alpaca 的模拟盘,也就是 paper trading,不涉及任何真实资金,也没有绑过任何银行卡。** 所有下单都发到 Alpaca 的模拟环境里。

技术栈很简单:Python 写核心逻辑,用 Alpaca 官方的 `alpaca-py` 做行情和下单,数据存在 SQLite,界面用 Streamlit。我们这套系统有个小亮点——**内置了两个可以随时切换的策略**,后面会演示。

接下来我先讲整体架构,然后我的队友会分别讲数据、策略、执行风控,最后我们现场把它跑起来给大家看。

## 1:00–3:00 · 整体架构(A)

[画面:README 里的架构图;然后切到编辑器,展开左边的文件夹树]

先看这张架构图。我们是完全**模块化**设计的,每一块各管一件事:

- 最上面是 **config**,所有参数——比如交易哪些股票、策略参数、风控上限——都写在一个 `config.yaml` 里;而 API 密钥这种敏感信息单独放在 `.env` 文件里,这个文件是被 git 忽略的,**绝不会提交到仓库**。
- 然后数据往下流:**data** 模块负责从 Alpaca 拉 K 线,存进 SQLite;
- **strategy** 模块把价格变成信号,我们约定信号就三种值:**+1 做多、0 空仓、-1 做空**;
- 信号先不直接下单,要先过 **risk** 风控模块,它负责算仓位大小、检查有没有超限;
- 通过之后交给 **execution** 执行模块,真正把订单发到 Alpaca,并且跟踪订单状态;
- 最后 **ui** 这个 Streamlit 面板,一边读账户实时状态、一边读 SQLite 里的日志,把一切展示出来,也能反过来控制引擎。

[画面:指一下文件夹树]

这里还有一个设计上我们比较满意的点:系统能跑两种模式——**回测**和**实盘模拟**——而且这两种模式**共用同一份策略和风控代码**。也就是说,你在回测里看到的逻辑,和实盘真正执行的逻辑是一模一样的,不会出现"回测一套、实盘另一套"的问题。**backtest** 模块就是把同样的策略拿到历史数据上重放。

下面请 B 讲数据管道和策略。

## 3:00–5:00 · 数据管道 + 策略(B)

[画面:打开 data/alpaca_client.py、fetcher.py,快速滑过]

我先讲数据。我们交易的是一篮子流动性很好的股票:苹果、微软、谷歌、亚马逊、英伟达、Meta、特斯拉,再加一个标普 500 的 ETF,一共 8 个标的。

`alpaca_client.py` 是我们对 Alpaca 官方 SDK 的一层封装,把账户、下单、行情这些接口收在一个地方,别的模块都不直接碰 SDK。`fetcher.py` 负责把 Alpaca 返回的多层索引数据整理成每个股票一张干净的 K 线表,再存进 SQLite。

[画面:config.yaml,指 feed: iex 那一行]

这里分享一个我们踩过的坑:免费的 Alpaca 账户只能用 **IEX** 这个数据源,如果用默认的 SIP 去拉最近的数据,会直接报"订阅权限不足"。所以我们在配置里显式设成了 `feed: iex`,这个细节卡了我们一会儿。

[画面:strategy/trend_ma.py]

再看策略。第一个是**趋势跟随**,用的是**双均线交叉**:快线默认 20 天、慢线 50 天,快线上穿慢线就做多,下穿就平掉或做空。**背后的直觉**是:价格是有趋势的,动量在中期会延续,所以我们顺着已经形成的趋势走。

[画面:strategy/mean_reversion.py]

第二个是**均值回归**,用价格相对滚动均值的 **z-score**:当价格异常便宜、z 低于 -2 个标准差,我们买入;异常贵、z 高于 +2,我们卖出;等它回到均值附近就平仓。**直觉**正好相反:短期里市场经常反应过度,然后会回归到平均水平。

这两个策略输出的都是刚才说的 +1 / 0 / -1 信号,接口完全统一,所以能随时切换。接下来 C 讲信号出来之后怎么控制风险、怎么下单。

## 5:00–7:00 · 执行 + 风控(C)

[画面:risk/risk_manager.py]

信号有了,但不能盲目下单,得先过风控。我们的风控做了这么几层:

- **单个股票最多占 15% 的权益**,不会把鸡蛋放一个篮子里;
- **总敞口不超过 1 倍权益**,也就是默认不加杠杆;
- 最多同时持有 10 个仓位;
- 每个持仓有 **5% 止损、10% 止盈**。

`risk_manager.py` 里,每一笔想下的单都会先在这里算好数量、再逐条检查有没有超过这些上限,超了就直接拦下来。

[画面:execution/engine.py 的 step() 方法]

真正的主循环在 `engine.py` 的 `step()` 里,一个周期干五件事:**拉最新数据 → 对每个股票算信号 → 检查现有持仓要不要止盈止损 → 把目标仓位和当前仓位的差额变成买卖单 → 全部记进日志**。

[画面:指 open_order_symbols 那段;或 order_manager.py]

这里我要特别讲一个我们真实遇到并修好的 bug——这也是我们对"真实交易系统"体会最深的地方。我们第一次跑的时候,订单全被 Alpaca 拒了,报错是 **"potential wash trade",潜在对敲交易**。查下来是两个原因叠加:一是当时**美股收盘了**,市价单只能排队不成交;二是我们的引擎每个周期都对同一只股票重下一遍,可上一单还挂着没成交,Alpaca 就判定这是重复对敲,拒单。

我们的修复是:**下单前先查一下这个股票有没有还没成交的挂单,有就跳过。** 我们还在面板上加了开/收盘状态提示和一键撤单。这个坑让我们真切理解到:真实交易里,市场时段、订单状态、交易所的合规检查,都是必须处理的东西,不是写个下单函数就完事了。

## 10:30–12:00 · 反思与总结(A / 全体)

[画面:README 的 Limitations 部分,或一张总结页]

最后聊聊局限和改进。

**局限方面:**
- 回测是理想化的——按收盘价成交,**没算手续费和滑点**,真实成交肯定会有偏差;
- 我们用的是免费的 IEX 数据源,它只覆盖一部分市场成交量,价格和全市场会有一点差异;
- 仓位管理比较简单,就是固定按权益百分比,没有根据波动率去动态调整;
- 界面是每次刷新驱动引擎跑一步,适合演示,但不是一个 7×24 常驻的进程。

**如果继续做,我们会:**
- 加入**波动率定仓**和组合层面的风险平价;
- 在回测里把**交易成本和滑点**建模进去;
- 用 **websocket 实时行情流**代替轮询;
- 做**滚动窗口的参数验证**,再加更多策略,比如因子或者机器学习。

**最后说说我们学到了什么。** 一开始我们以为难点是策略,真做下来才发现,让一个交易系统**稳定地跑起来**才是最花功夫的:要处理市场开收盘、订单的各种状态、交易所的合规拒单、密钥怎么安全管理、回测和实盘怎么共用一套代码。这些"工程"上的事,才是真实交易系统和课堂练习最大的区别。
